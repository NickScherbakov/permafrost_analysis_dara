# Permafrost Analysis

This repository contains tools and scripts for analyzing permafrost data. The aim is to facilitate research and provide insights into permafrost conditions and dynamics.

## Математическая модель, реализованная в notebook

Основная реализация находится в `permafrost_analysis.ipynb`. Модель не является черным ящиком (типа нейросети), а представляет собой физико-статистическую систему индикаторов с прозрачными правилами принятия решений.

### 1. Цель модели

Построить воспроизводимую цепочку анализа:

климатические аномалии -> деградация мерзлоты -> гидрологические риски -> связь с ЧС -> сценарий реагирования.

Модель работает на трех горизонтах:

- суточный и месячный (оперативные сигналы),
- годовой синхронный (год N -> ЧС в году N),
- годовой лаговый (год N -> ЧС в году N+1).

### 2. Входные данные

Используются ряды из `data/`:

- температура воздуха и осадки (TTTR),
- температура почвы,
- снежный покров,
- уровни воды по гидропостам Лены,
- глубина сезонного протаивания (CALM, ALT),
- события ЧС (МЧС).

Базовый период анализа для основной части расчётов: 2008-2023.

### 3. Физические индикаторы

#### 3.1. Индексы тепла/холода

Для каждой станции рассчитываются:

- `TDD` (thawing degree days): сумма положительных среднесуточных температур за апрель-октябрь
- `FDD` (freezing degree days): сумма модулей отрицательных температур за зимний сезон

Формально:

`TDD(y) = sum(max(Tmean(d), 0))`, `d in Apr..Oct, year=y`

`FDD(y) = sum(max(-Tmean(d), 0))`, `d in winter_season(y)`

Важная деталь: зима задается как сквозной сезон (ноябрь-декабрь прошлого года + январь-март текущего), чтобы правильно отражать физику паводкового года.

#### 3.2. Зимние и осадочные индикаторы

- `winter_thaw_days`: число зимних дней, где `Tmean > 0`
- `max_daily_precip`: максимум суточных осадков за год
- `spring_precip`: сумма осадков в апреле-мае
- `max_snow_depth`: максимум высоты снега за сквозной снежный сезон

#### 3.3. Rain-on-Snow (ROS)

Событие ROS детектируется, если одновременно выполняются условия:

- месяц в октябре-апреле,
- `Tmean > 0`,
- `precip_mm > 1`,
- `snow_depth_cm > 5`.

Для годового учета используется физический сезон:

- октябрь-декабрь N относятся к сезону N+1,
- январь-апрель N относятся к сезону N.

#### 3.4. Мерзлотные параметры

Вычисляются:

- `MAGST`: средняя годовая температура грунта (в коде используется глубина 20 см),
- `TTOP`: температура на кровле мерзлоты (160 см, fallback 320 см),
- `MAAT`: средняя годовая температура воздуха,
- `SO = MAGST - MAAT` (surface offset),
- `TO = TTOP - MAGST` (thermal offset),
- `N_t = TDD_soil / TDD_air`,
- `N_f = FDD_soil / FDD_air`.

### 4. Детекция аномалий

Для каждого показателя применяется комбинированная схема:

1. Глобальный Z-score:

`z = (x - mean(x)) / std(x)`

2. Перцентильные пороги:

- умеренная аномалия: верхний хвост `P95`,
- экстремальная: `P99`.

3. Локальная rolling-норма:

- окно 7 лет,
- минимум 4 прошлых года,
- текущий год исключается из базовой нормы через `shift(1)`.

Для индикаторов, где "меньше = хуже" (например, `FDD`), вектор перед нормировкой инвертируется. Это приводит все признаки к единой логике: "больше значения аномальности = выше риск".

Итоговые флаги `moderate` и `extreme` формируются как логическое OR по трем критериям (global Z, percentile, rolling Z).

### 5. Оперативная модель (дни и месяцы)

#### 5.1. Суточные сигналы

Строится суточная климатология по дню года (DOY) с окном +/-7 дней. Затем для каждой даты определяются сигналы:

- аномально теплый день (`tmean_z >= 2`),
- зимняя/весенняя оттепель (`month in Oct..Apr`, `tmean > 0`, `tmean_z >= 1.5`),
- экстремальные суточные осадки (`precip >= P95`),
- резкое потепление (`delta_tmean_1d >= 8 C`).

Считаются `risk_points`, затем присваивается `severity`:

- "экстремальная" при сильных порогах (например, `tmean_z >= 3`, `precip >= P99` или высокий суммарный риск),
- иначе "умеренная".

#### 5.2. Месячные сигналы

Месячные агрегаты сравниваются с нормой того же календарного месяца в другие годы:

- аномально теплый месяц,
- аномально влажный месяц,
- аномально много дней зимней оттепели.

### 6. Гидрологический подмодуль

По каждому гидропосту рассчитываются:

- `spring_peak_cm`,
- `spring_amplitude_cm`,
- `max_rise_7d_cm_day`,
- `days_above_p90`, `days_above_p95`,
- Z-оценки этих признаков.

Также строятся:

- лаги дат весеннего пика относительно Якутска,
- доля гидропостов с гидроэкстремами,
- корреляция этой доли с числом ЧС по годам.

### 7. Связь аномалий с ЧС

Используются два уровня связывания.

#### 7.1. Синхронный годовой анализ

Сопоставление индикаторов года N с `chs_count` года N.

Годы делятся на классы:

- `safe`: 0 ЧС,
- `norm`: 1 ЧС,
- `spike`: >1 ЧС.

Для каждого Z-признака рассчитывается контрастный сдвиг:

`delta_z = mean(z | spike) - mean(z | safe)`

#### 7.2. Лаговый годовой анализ

Индикаторы года N сопоставляются с ЧС года N+1.

Технически это реализовано через сдвиг:

`year_chs = year_indicators + 1`

И далее применяется тот же контрастный анализ по классам `safe/norm/spike`.

### 8. Сценарная модель реагирования

На основе количества сработавших годовых флагов:

- `n_moderate`,
- `n_extreme`,
- при наличии данных CALM учитывается `ALT_z`,

назначается сценарий:

- Базовый,
- Умеренный,
- Высокий,
- Критический.

Каждому сценарию сопоставлен готовый набор мер реагирования (мониторинг, оповещение, готовность МЧС, эвакуационные действия и т.д.).

### 9. Что модель даёт на практике

Модель дает интерпретируемый ответ:

- какие именно физические индикаторы вышли в аномалию,
- насколько аномалия сильна (Z-score, перцентиль, rolling-отклонение),
- есть ли подтверждение через лаговую/синхронную связь с ЧС,
- какой сценарий реагирования активируется.

Это делает систему пригодной не только для научной интерпретации (ВКР), но и для прикладного протокола раннего предупреждения.

### 10. Ограничения и допущения

- Пороги заданы экспертно (rule-based), не через обучаемую оптимизацию.
- Период 2008-2023 сравнительно короткий для редких экстремумов.
- Качество связи "аномалия -> ЧС" зависит от полноты и точности базы ЧС.
- Часть выводов чувствительна к пространственному покрытию станций и постов.

При этом ключевое преимущество реализации - прозрачность, воспроизводимость и физическая интерпретируемость каждого шага.
